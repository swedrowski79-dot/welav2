# Ticket: T-030

## Status
done

## Title
Implement full XT-Commerce writer for product export (end-to-end sync)

## Problem
The system already calculates product deltas and queues product export entries.
However, there is no complete XT-Commerce writer that fully synchronizes products (including base data, translations, attributes, media, and documents) into the shop.

## Goal
Implement a complete product export pipeline into XT-Commerce using the existing queue, delta, and state system.

## Scope
- consume `export_queue` rows with:
  - `entity_type = product`
- map product payloads to XT-Commerce structure:
  - base product data
  - translations
  - attributes (if available)
  - categories (existing linkage)
- ensure compatibility with:
  - media (T-029)
  - documents (T-029)
- ensure full product consistency in XT

## Requirements
- create or update products in XT-Commerce
- update existing products without duplication
- write translations correctly
- link products to categories correctly
- integrate with existing media/document export (T-029)
- update confirmed export state only after successful write
- handle partial failures safely

## Acceptance Criteria
- [x] product queue rows are written into XT-Commerce
- [x] updates correctly modify existing products
- [x] no duplicate products are created
- [x] translations are written correctly
- [x] category assignments are correct
- [x] media and documents are linked after export
- [x] queue rows move to `done` only after success
- [x] failed exports are marked as `error`
- [x] confirmed export state is consistent

## Files / Areas
- `config/xt_write.php`
- XT writer service(s)
- `src/Service/ExportQueueWorker.php`
- product mapping layer
- translation handling
- category mapping
- monitoring/logging

## Notes
This ticket completes the full system:
AFS → Pipeline → Delta → Queue → XT-Commerce

Keep the implementation incremental and consistent with existing patterns.
Do not redesign delta, queue, or state logic.

## Implementation Notes
- Added a dedicated `XtProductWriter` and an `XtCompositeWriter` so product, media, and document exports all use the same `ExportQueueWorker` entrypoint while keeping entity-specific logic focused.
- Extended `config/xt_write.php` for current stage payload reality:
  - product translations now map from `translation.*`
  - category relations carry explicit replace metadata
  - XT attribute entity/description/link definitions were added for product attributes
- Added a transactional `sync_product` action to `wela-api` so one product export can atomically write:
  - `xt_products`
  - `xt_products_description`
  - `xt_products_to_categories`
  - `xt_plg_products_attributes`
  - `xt_plg_products_attributes_description`
  - `xt_plg_products_to_attributes`
- Product export state is still confirmed only after the XT write succeeds; category lookup failures or other write errors leave the queue row in `error` and do not advance `product_export_state`.
