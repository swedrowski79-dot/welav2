# Ticket: T-029

## Status
done

## Title
Implement XT-Commerce writer for product media and document export queue entries

## Problem
The pipeline now imports, normalizes, stages, calculates delta, and queues product media and document changes correctly.
However, there is still no writer that consumes `media` and `document` queue entries and writes them into XT-Commerce.

## Goal
Implement the XT-Commerce write layer for media and document queue entries so staged product-linked files can be exported into the target shop system.

## Scope
- consume `export_queue` rows with:
  - `entity_type = media`
  - `entity_type = document`
- map queue payloads to the existing XT write configuration and target tables
- create/update XT media/document records
- create/update the product-to-media / product-to-document relations in XT
- keep success confirmation aligned with the existing export worker flow
- do not redesign the existing queue, delta, or state architecture

## Acceptance Criteria
- [x] media queue rows can be written into XT-Commerce successfully
- [x] document queue rows can be written into XT-Commerce successfully
- [x] product linkage is written correctly in XT
- [x] successful writes mark queue rows as `done`
- [x] failed writes mark queue rows as `error`
- [x] confirmed export state is only updated after successful XT write
- [x] repeated runs do not create duplicate active exports for the same media/document entity

## Files / Areas
- `config/xt_write.php`
- XT writer service(s)
- `src/Service/ExportQueueWorker.php`
- media/document export mapping layer
- related monitoring/logging files if needed

## Notes
Keep the XT integration incremental and safe.
Do not change product/core delta logic in this ticket.
Prefer reuse of existing queue/worker patterns over introducing a separate export architecture.

## Implementation Notes
- `run_export_queue.php` now wires a focused `XtMediaDocumentWriter` into the existing `ExportQueueWorker`, while product entries keep the previous local confirmation flow.
- Media/document queue payloads are mapped through `config/xt_write.php` and resolved against XT product/media identities via the `wela-api` transport instead of direct XT DB access.
- `wela-api/` now exposes minimal safe actions for `lookup_map`, `upsert_row`, and `delete_rows`, which are sufficient for media/document entity upserts and link maintenance.
- Relation configs in `config/xt_write.php` were extended with queue-safe identity/delete metadata so repeated worker runs stay idempotent and removal payloads can unlink via `m_id` + `type`.
